# Yamibo App Updater Release Guide

This document describes the manifest used by the in-app updater and the Android APK release flow.

## Manifest

The app checks these manifests in order:

1. `https://raw.githubusercontent.com/LittleSurvival/yamibo-app/update-release/update/stable.json`
2. `https://gitee.com/LittleSurvival/ymb-apk-release/raw/main/update/stable.json`
3. `https://gitcode.com/LittleSurvival/ymb-apk-release/raw/main/update/stable.json`

Keep the same published `update/stable.json` file in each mirror. The Gitee/GitCode URLs are intended for users who cannot reliably reach GitHub.
`update/manifest.json` is the manually edited source of truth in the source repo. `update/stable.json` is generated by `syncStableManifest`.
Source manifests must always keep `isReady=false`; published manifests are generated by the release workflow after the APK has been uploaded.

```json
{
  "channel": "stable",
  "versionName": "1.2.0",
  "versionCode": 120,
  "isReady": false,
  "minVersionCode": 100,
  "releaseNotes": "Fixes and improvements.",
  "assets": []
}
```

Fields:

- `isReady`: source manifests must be `false`. The release workflow publishes `true` only after APK upload succeeds.
- `versionCode`: the numeric value compared with the installed Android `versionCode`. Increase this for every release.
- `versionName`: display version shown to users.
- `minVersionCode`: optional lower bound for future forced-update policy. The app records it now but does not force updates yet.
- `releaseNotes`: plain text shown in the update screen.
- `releaseUrl`: published-only field opened when the user taps the release-page action.
- `assets`: published-only installable files. Android currently accepts `type = "universal-apk"` or `"apk"`.
- `sha256`: required for production releases. The app verifies it before opening the installer.
- `size`: optional byte count used by the progress UI.

## Android APK Release Flow

1. Update `versionCode` and `versionName` in `composeApp/build.gradle.kts`.
2. Update `update/manifest.json`.
3. Run manifest sync and validation:

```powershell
.\gradlew syncStableManifest validateUpdateManifest --console=plain
```

The build fails if `composeApp/build.gradle.kts`, `update/manifest.json`, `update/stable.json`, and `update/changelogs/{versionCode}.changelog` are inconsistent.
It also fails if source manifests set `isReady=true`, contain `releaseUrl`, or contain non-empty `assets`.

Manual APK flow:

1. Build a release APK:

```powershell
.\gradlew :composeApp:assembleRelease --console=plain
```

3. Sign the APK if the build is not already configured with a release signing config.
4. Calculate SHA-256:

```powershell
Get-FileHash .\composeApp\build\outputs\apk\release\composeApp-release.apk -Algorithm SHA256
```

5. Upload the APK to GitHub Releases.
6. Generate a published update folder with `isReady=true`, `releaseUrl`, and APK `assets`.
7. Validate it with `validatePublishedUpdateManifest`.
8. Push the published update folder to the GitHub `update-release` branch and the Gitee/GitCode mirror repos.
9. Open the app Settings -> App updates -> Check update.

## iOS Update Flow

iOS cannot install an IPA directly from inside a normal App Store/TestFlight app. The app updater therefore only checks the manifest and opens `releaseUrl`.

Recommended options:

- App Store: use an App Store product URL as `releaseUrl`.
- TestFlight: use the TestFlight invite/app URL.
- Enterprise/MDM/internal distribution: use the organization's HTTPS landing page. The actual install must be handled by Safari, MDM, or Apple's enterprise install flow.
